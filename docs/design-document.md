# AI 视觉对话助手 —— 设计文档

## 1. 项目概述

AI 视觉对话助手是一款能够让用户通过摄像头与麦克风与 AI 进行自然对话的应用。AI 能够实时"看见"摄像头画面中的内容，并基于视觉信息与用户的语音/文字输入给出恰当的回应。

### 1.1 技术架构

```
┌─────────────────────────────────────────────────────┐
│                  浏览器 (前端)                        │
│  ┌──────────┐   ┌───────────┐   ┌────────────────┐  │
│  │ getUserMedia│ │ Web Speech │   │ SpeechSynthesis│  │
│  │ 摄像头捕获  │ │ API (STT) │   │ API (TTS)     │  │
│  └─────┬────┘   └─────┬─────┘   └───────┬────────┘  │
│        │ Canvas帧捕获  │ 语音→文本       │ 文本→语音   │
│        ▼               ▼                 │           │
│  ┌───────────────────────────────────────┘           │
│  │  HTTP POST /api/chat                              │
│  │  { text, image(base64), conversation_history }    │
│  └──────────────────────┬───────────────────────────┘│
└─────────────────────────┼────────────────────────────┘
                          │
┌─────────────────────────┼────────────────────────────┐
│              Python FastAPI (后端)                    │
│  ┌──────────────────────▼──────────────────────────┐ │
│  │  /api/chat:                                      │ │
│  │    1. 图像压缩 (640×480, JPEG 60%)               │ │
│  │    2. 构建消息 (System + History + Current)       │ │
│  │    3. 调用 DeepSeek API                           │ │
│  │    4. 返回 AI 回复                                │ │
│  └──────────────────────┬──────────────────────────┘ │
└─────────────────────────┼────────────────────────────┘
                          │
┌─────────────────────────┼────────────────────────────┐
│              DeepSeek-VL2 API (云端视觉大模型)        │
│              ¥1/百万 tokens                          │
└──────────────────────────────────────────────────────┘
```

### 1.2 核心流程

1. **摄像头**: 用户开启摄像头，浏览器通过 `getUserMedia` 获取视频流并预览
2. **用户发言**: 按住说话按钮（PTT），Web Speech API 将语音转为文本
3. **帧捕获**: 在发送请求前，从 Canvas 截取当前帧，压缩为 JPEG Base64
4. **AI 请求**: 前端将文本 + 图片 + 对话历史发送给后端 `/api/chat`
5. **后端处理**: 后端压缩图片、构建消息、调用 DeepSeek-VL2 API
6. **语音回复**: 后端返回文本，前端通过 SpeechSynthesis API 播报

## 3. 运营成本控制策略

### 3.1 识别的成本构成

以 DeepSeek-VL2 API 为例（¥1/百万 tokens），每次对话的成本构成：

```
单次请求成本 = 输入 tokens × ¥1/百万 + 输出 tokens × ¥1/百万
```

其中输入 tokens 包含：
- System prompt: ~100 tokens
- 用户文本: ~10-50 tokens
- 图片: ~258-500 tokens (根据图片分辨率)
- 历史对话: O(n) tokens

### 3.2 考虑过的控制策略

| # | 策略 | 预计节省 | 是否实施 | 未实施原因 |
|---|------|---------|---------|-----------|
| 1 | **按需帧捕获**：仅在用户发送消息时捕获一次帧 | 减少 90% 视觉 API 调用 | **✅ 是** | — |
| 2 | **图像压缩**：缩放到 640×480，JPEG 60% | 图像从 ~200KB→~50KB | **✅ 是** | — |
| 3 | **本地 STT/TTS**：Web Speech API 零费用 | 节省 ¥0.03-0.10/次 | **✅ 是** | — |
| 4 | **System Prompt 控长**：要求回答 ≤2 句 | 输出 tokens 减少 60%+ | **✅ 是** | — |
| 5 | **上下文裁剪**：仅保留最近 10 轮 | 避免长上下文累积 | **✅ 是** | — |
| 6 | **运动检测**：仅画面变化时发送帧 | 再减少 ~30% 调用 | **❌ 否** | 复杂度高，计算也在客户端，收益有限 |
| 7 | **图像差分**：只发送画面变化区域 | 减少传输量 ~50% | **❌ 否** | 实现复杂度高，与 API 格式不兼容 |
| 8 | **缓存常见回应**：对常见场景的回复缓存 | 省去重复 API 调用 | **❌ 否** | 视觉场景多变，缓存命中率低 |
| 9 | **模型降级**：对简单问题自动用更便宜模型 | 可省 50%+ | **❌ 否** | 同场景内切换模型体验割裂 |
| 10 | **批处理请求**：合并短时间内的多次请求 | 减少 HTTP 开销 | **❌ 否** | 实时对话场景下不适用 |

### 3.3 实际采用的策略及量化效果

| # | 策略 | 实现位置 | 量化效果预估 |
|---|------|---------|-------------|
| 1 | **按需帧捕获** | `app.js:getFrameBase64()` — 仅在发送时捕获 | 若持续对话 10 分钟（~20 轮），仅发送 20 帧 vs 持续 1fps 的 600 帧 → 减少 96.7% |
| 2 | **图像压缩** | `backend/main.py:compress_image()` — 640×480, JPEG 60%, LANCZOS | 单帧从 ~200-500KB 降至 ~30-80KB（~80%），对应 tokens 从 ~500+ 降至 ~258 |
| 3 | **本地 STT** | `app.js:initSpeechRecognition()` — Web Speech API | 每次语音识别节省 ¥0.03-0.10（对比云 STT 如阿里/腾讯） |
| 4 | **本地 TTS** | `app.js:speakText()` — SpeechSynthesis API | 每次语音合成节省 ¥0.02-0.08（对比云 TTS） |
| 5 | **System Prompt 控长** | `backend/main.py` — "回答控制在 2 句话以内" | 每次回复平均 ~80 tokens vs 无限制的 ~300 tokens → 节省 ~73% |
| 6 | **上下文裁剪** | `app.js:sendToAI()` + `backend/main.py` — 各保留 10 轮 | 最多 20 轮对话上下文，避免无限累积 |
| 7 | **max_tokens 限制** | `backend/main.py` — `max_tokens=300` | 单次回复最多 300 tokens，防止异常长回复 |

### 3.4 综合成本估算

**假设**：每天 100 次对话，每次对话 3 轮（1 问 1 答），每轮 1 张图

| 项目 | 无优化 | 优化后 | 节省 |
|------|--------|--------|------|
| 图像 tokens/次 | 500 | 258 | 48% |
| 输出 tokens/次 | 300 | 80 | 73% |
| API 调用费用/天 | ¥0.46 | ¥0.18 | 61% |
| 语音识别费用/天 | ¥3.00 | ¥0.00 | 100% |
| 语音合成费用/天 | ¥2.00 | ¥0.00 | 100% |
| **总计/天** | **¥5.46** | **¥0.18** | **96.7%** |
| **总计/月** | **¥164** | **¥5.4** | **96.7%** |

> 注：DeepSeek 价格为 ¥1/百万 tokens，STT/TTS 对比阿里云/腾讯云公开定价。
> 实际费用因使用模式而异。按需帧捕获的节省主要体现在调用次数而非单次费用。

## 4. 关键设计决策

### 4.1 为什么选择 Web Speech API 而不是云 STT/TTS？

| 维度 | Web Speech API (浏览器原生) | 云 STT/TTS (阿里/腾讯/百度) |
|------|---------------------------|---------------------------|
| 成本 | **免费** | ¥0.01-0.03/次 |
| 延迟 | <100ms（本地处理） | 200-500ms（网络延迟） |
| 精度 | 中文识别率约 90-95% | 中文识别率约 95-98% |
| 离线 | 需网络（引擎在云）| 需网络 |
| 集成 | 零配置 | 需 SDK/API Key |

**决策**：优先采用 Web Speech API。对于个人/小团队场景，免费和低延迟的价值超过了精度的小幅差距。

### 4.2 为什么选择按需帧捕获而不是持续捕获？

按需捕获（在用户发送消息时才截帧）vs 持续捕获：

- **持续 1fps 捕获**：用户 10 分钟产生 600 帧，但只有 20 次真实对话
- **按需捕获**：20 次对话仅截取 20 帧 → 减少 96.7% 的无效数据
- **实时性**：按需捕获在按下发送按钮到请求发出之间仅延迟 ~50ms，用户无感知

**决策**：采用按需捕获 + 手动快照功能作为补充。

### 4.3 为什么选择 Python FastAPI？

| 维度 | Python FastAPI | Node.js Express | Go |
|------|---------------|-----------------|-----|
| 图像处理 | Pillow 成熟可靠 | sharp 性能好但生态略弱 | 需额外库 |
| HTTP 客户端 | httpx (async) | 内置 fetch | 需额外库 |
| AI API 兼容性 | 最广的生态 | 良好 | 一般 |
| 开发速度 | 快速 | 快速 | 较慢 |
| 部署简便性 | pip install + python main.py | npm install + node index.js | 需编译 |

**决策**：Python FastAPI 在图像处理（Pillow）和 AI API 集成方面生态最为成熟，且配置部署简单，适合原型开发到生产部署的全周期。

### 4.4 为什么选择 DeepSeek-VL2？

| 模型 | 价格 (输入) | 视觉能力 | 中文能力 | 开发者体验 |
|------|------------|---------|---------|-----------|
| **DeepSeek-VL2** | **¥1/百万 tokens** | ★★★★☆ | ★★★★★ | ★★★★★ (OpenAI 兼容) |
| GPT-4o | ¥18/百万 tokens | ★★★★★ | ★★★★☆ | ★★★★★ |
| Qwen-VL-Max | ¥4/百万 tokens | ★★★★☆ | ★★★★★ | ★★★★☆ |
| GLM-4V | ¥5/百万 tokens | ★★★★☆ | ★★★★★ | ★★★★☆ |

**决策**：DeepSeek-VL2 在性价比上具有压倒性优势（¥1/百万 tokens），视觉能力满足需求，且 API 完全兼容 OpenAI 格式，适配成本最低。

## 5. 未来优化方向

### 5.1 功能扩展
- **连续对话模式**：识别到静音后自动重新开始监听，无需持续按住按钮
- **音频电平可视化**：用 AudioContext 分析实时音量，在 UI 显示音频波形
- **多模态输出**：AI 不仅能说，还能在画面中标注/框出识别到的物体
- **屏幕共享**：增加屏幕共享模式，用于演示/教学场景

### 5.2 性能优化
- **Web Worker 帧处理**：图像缩放和压缩在 Worker 线程执行，不阻塞主线程
- **IndexedDB 缓存**：缓存常见场景的 AI 回复，减少 API 调用
- **流式响应**：SSE 流式输出 AI 回复，实现打字机效果

### 5.3 成本优化
- **图像差分检测**：通过像素级差分检测场景变化，仅发送变化的画面区域
- **模型自动降级**：简单问答用文本模型，涉及视觉的再用视觉模型
- **本地小模型**：在端侧运行轻量视觉模型做预处理过滤

## 6. 部署说明

### 6.1 环境要求
- Python 3.10+
- 现代浏览器（Chrome/Edge 最新版）
- DeepSeek API Key（[platform.deepseek.com](https://platform.deepseek.com)）

### 6.2 启动命令
```bash
cd backend
pip install -r requirements.txt
# 配置 .env 文件: DEEPSEEK_API_KEY=sk-xxx
python main.py
# 打开 http://localhost:8000
```

### 6.3 注意事项
- 首次使用时浏览器会请求摄像头和麦克风权限，需允许
- 如使用 HTTPS 部署，Web Speech API 在安全上下文下工作更稳定
- 生产部署建议使用 `uvicorn` 配合反向代理（Nginx）
